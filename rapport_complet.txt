1. INTRODUCTION ET PROBLEMATIQUE
================================================================================

1.1 Contexte General
--------------------
Les reseaux 5G representent une revolution dans les telecommunications.
Contrairement a la 4G qui offrait un service unique "best effort" (au mieux),
la 5G introduit le concept de "Network Slicing" : la possibilite de decouper
un meme reseau physique en plusieurs reseaux virtuels, chacun avec ses propres
garanties de qualite de service.

Par exemple, sur la MEME antenne 5G :
- Un chirurgien peut operer a distance (besoin de latence ultra-faible)
- Un utilisateur peut regarder Netflix en 4K (besoin de haut debit)
- 10 000 capteurs IoT peuvent envoyer des donnees (besoin de connexion massive)

Chaque "slice" (tranche) du reseau a un contrat de qualite appele SLA
(Service Level Agreement). Si ce contrat est viole, les consequences peuvent
etre graves : une operation chirurgicale ratee, une video qui saccade, etc.

1.2 Question de Recherche
--------------------------
    Comment PREDIRE les violations SLA AVANT qu'elles ne se produisent,
    pour permettre au reseau de reagir de maniere proactive ?

L'idee est d'utiliser l'intelligence artificielle (un reseau de neurones LSTM)
pour analyser les metriques du reseau en temps reel et alerter 15 secondes
avant qu'une violation ne se produise.

1.3 Deux Approches Comparees
-----------------------------
Ce PFE explore deux approches complementaires :

    APPROCHE A : Modele Mathematique (M/M/1)
    - Utilise des formules de files d'attente pour ESTIMER la latence
    - Rapide a calculer mais simplifie (ignore la radio, le scheduling...)
    - Entraine sur des donnees reelles (captures Wireshark GeForce Now)

    APPROCHE B : Simulation Realiste (simu5G)
    - Simule TOUT le reseau 5G dans un ordinateur (canal radio, protocoles...)
    - Latence MESUREE et non estimee
    - Permet de creer des scenarios de stress controles
    - Entraine sur des donnees simulees (13 scenarios differents)

2. CONCEPTS FONDAMENTAUX (pour non-specialistes)
================================================================================

2.1 Qu'est-ce que la 5G ?
--------------------------
La 5G est la 5eme generation de reseau mobile. Ses promesses principales :

    +------------------+------------------+-------------------+
    |   Debit (vitesse) | Latence (delai)  | Connexions        |
    +------------------+------------------+-------------------+
    | 4G : ~100 Mbps   | 4G : ~30-50 ms   | 4G : ~100k/km2   |
    | 5G : ~1-10 Gbps  | 5G : ~1-10 ms    | 5G : ~1M/km2     |
    +------------------+------------------+-------------------+

La 5G n'est pas juste "de la 4G plus rapide". C'est une architecture
completement nouvelle, definie par le consortium 3GPP (3rd Generation
Partnership Project), qui regroupe les operateurs et fabricants mondiaux.

2.2 Qu'est-ce que le Network Slicing ?
----------------------------------------
Imaginez une autoroute avec 6 voies. Au lieu que toutes les voitures
se melangent, on reserve :
- 2 voies pour les ambulances (URLLC - urgences, latence ultra-faible)
- 3 voies pour les camions (eMBB - gros volumes, haut debit)
- 1 voie pour les velos (mMTC - plein de petits objets connectes)

Chaque "voie reservee" est un SLICE. Les 3 types standard (3GPP TS 23.501) :

    +----------+-------------------------------+------------------------+
    |  Slice   | Description                   | Exemple Concret        |
    +----------+-------------------------------+------------------------+
    |  eMBB    | Enhanced Mobile Broadband     | Streaming video 4K,    |
    |          | = Haut debit                  | cloud gaming, VR       |
    +----------+-------------------------------+------------------------+
    |  URLLC   | Ultra-Reliable Low-Latency    | Chirurgie a distance,  |
    |          | = Faible latence, fiable      | conduite autonome      |
    +----------+-------------------------------+------------------------+
    |  mMTC    | Massive Machine-Type Comms    | Capteurs IoT, compteurs|
    |          | = Connexion massive d'objets  | intelligents, agriculture|
    +----------+-------------------------------+------------------------+

2.3 Qu'est-ce qu'un SLA ?
---------------------------
SLA = Service Level Agreement = Contrat de qualite de service.

C'est un engagement chiffre de l'operateur envers le client. Par exemple :

    +----------+---------------------------+---------------------------+
    |  Slice   | Metrique                  | Seuil SLA                 |
    +----------+---------------------------+---------------------------+
    |  eMBB    | Debit minimum             | >= 10 Mbps                |
    |          | Latence maximum           | <= 50 ms                  |
    |          | Perte de paquets max      | <= 1%                     |
    +----------+---------------------------+---------------------------+
    |  URLLC   | Debit minimum             | >= 0.5 Mbps               |
    |          | Latence maximum           | <= 20 ms                  |
    |          | Perte de paquets max      | <= 0.001%                 |
    +----------+---------------------------+---------------------------+
    |  mMTC    | Debit minimum             | >= 0.0003 Mbps            |
    |          | Latence maximum           | <= 8.8 ms                 |
    |          | Perte de paquets max      | <= 1%                     |
    +----------+---------------------------+---------------------------+

Si UNE SEULE de ces metriques depasse son seuil, le SLA est considere VIOLE.

2.4 Pourquoi predire les violations ?
---------------------------------------
Aujourd'hui, les operateurs detectent les violations APRES qu'elles se
produisent. C'est trop tard : le chirurgien a deja eu un retard, le joueur
a deja subi un lag.

Notre objectif : predire la violation 15 secondes a l'avance, pour que le
reseau puisse reagir :
- Reallouer des ressources (donner plus de "voies" au slice en difficulte)
- Devier le trafic vers une autre antenne
- Prevenir l'utilisateur (qualite degradee)

5. APPROCHE A : MODELE ANALYTIQUE M/M/1 + LSTM
================================================================================

5.1 Le Modele M/M/1 Explique Simplement
-----------------------------------------
Le modele M/M/1 est un modele mathematique de file d'attente.

    Imaginez un bureau de poste avec UN SEUL guichet :
    - Les clients arrivent de facon aleatoire (en moyenne lambda par heure)
    - Chaque client est servi en un temps aleatoire (en moyenne 1/mu)
    - Il n'y a qu'un guichet (1 serveur)

    +---+---+---+---+---+---------+
    | 5 | 4 | 3 | 2 | 1 | GUICHET |----> Sortie
    +---+---+---+---+---+---------+
         File d'attente     Serveur

    Formules fondamentales :
    - rho = lambda / mu           (taux d'occupation, ex: 80%)
    - W = 1 / (mu - lambda)       (temps moyen dans le systeme = LATENCE)
    - Wq = rho / (mu - lambda)    (temps d'attente en file uniquement)

    Exemple concret :
    Si lambda = 1000 paquets/s et mu = 1200 paquets/s
    rho = 1000/1200 = 0.83 (83% d'occupation)
    W = 1/(1200-1000) = 5 ms de latence moyenne

    Quand rho approche 1 (saturation), la latence EXPLOSE.
    C'est ce phenomene qui cause les violations SLA.

5.2 Application au Network Slicing
------------------------------------
Pour notre PFE, chaque slice est modelise comme une file M/M/1 independante :

    +---------+    lambda = throughput_mesuree
    | Slice   |    mu = capacite_du_slice
    | eMBB    |    W = 1/(mu - lambda) = LATENCE ESTIMEE
    +---------+

    Capacites supposees (par slice) :
    - eMBB     : 100 Mbps
    - URLLC    : 10 Mbps
    - mMTC     : 1 Mbps
    - BestEffort : 50 Mbps

    LIMITE IMPORTANTE :
    Le M/M/1 ne modelise PAS le canal radio 5G, ni le scheduling,
    ni les protocoles (PDCP, RLC, MAC, PHY), ni les retransmissions.
    La latence estimee est donc APPROXIMATIVE.

5.3 Pipeline de Donnees Approche A
------------------------------------
    GeForce_Now_*.csv (9 fichiers, 7 GB)
            |
            v
    [Apache Spark - slices.py]

               Jitter, PacketLoss, NetworkLoad, SLA_Violated_in_15s
            |

    - Moyennes mobiles (5s, 10s) pour capturer les tendances

5.4 Architecture du LSTM (Approche A)
---------------------------------------
LSTM = Long Short-Term Memory = Un type de reseau de neurones specialise
pour les donnees temporelles (series chronologiques).

    Pourquoi un LSTM et pas un simple reseau de neurones ?
    Parce que les violations SLA dependent de l'HISTORIQUE :
    "Si le debit baisse depuis 20 secondes, une violation est probable."
    Le LSTM a une MEMOIRE qui retient le passe.

    Architecture detaillee :

    Entree : 30 pas de temps x 20 features
    (= les 30 dernieres secondes de metriques reseau)
        |
        v
    +-------------------------------+
    | LSTM Couche 1 (64 neurones)   |  Apprend les patterns temporels
    | + Dropout 30%                 |  Le dropout evite le surapprentissage
    +-------------------------------+
        |
        v
    +-------------------------------+
    | LSTM Couche 2 (32 neurones)   |  Raffine les patterns
    | + Dropout 30%                 |
    +-------------------------------+
        |
        v
    +-------------------------------+
    | Couche Dense (32 -> 16)       |  Combine les features
    | Activation ReLU               |
    +-------------------------------+
        |
        v
    +-------------------------------+
    | Couche de Sortie (16 -> 1)    |  Prediction binaire
    | Activation Sigmoid            |  0 = OK, 1 = Violation
    +-------------------------------+

    Nombre total de parametres : 35 105

    Entrainement :
    - Fonction de perte : BCEWithLogitsLoss (classification binaire)
    - Poids de classe (pos_weight=1.78) pour compenser le desequilibre
      (35% violations vs 65% OK)
    - Optimiseur : Adam (learning rate = 0.001)
    - Scheduler : Reduit le learning rate si la performance stagne
    - Batch size : 256, Epochs : 50 (arret anticipe a l'epoch 47)

6. APPROCHE B : SIMULATION simu5G + LSTM
================================================================================

6.1 Qu'est-ce que simu5G ?
----------------------------
simu5G est un simulateur open-source developpe par l'Universite de Pise
(Italie) qui permet de simuler des reseaux 5G complets.

    simu5G = Extension de OMNeT++ + INET pour la 5G

    OMNeT++ : Moteur de simulation a evenements discrets
    - Comme un "jeu video" du reseau : chaque paquet est simule
    - Le temps avance d'evenement en evenement
    - On peut simuler 1 heure de reseau en quelques minutes

    INET Framework : Bibliotheque de protocoles reseau
    - IP, UDP, TCP, Ethernet...
    - Fournit les briques de base (Router, StandardHost)

    simu5G : Composants 5G
    - gNodeB (antenne 5G), NrUe (terminal mobile), Upf (routeur 5G)
    - Simule le canal radio 5G NR (propagation, fading, interference)
    - Implemente les protocoles 3GPP (PDCP, RLC, MAC, PHY)

    Publication de reference :
    G. Nardini, G. Stea, A. Virdis, "Simu5G - An OMNeT++ Library for
    End-to-End Performance Evaluation of 5G Networks", IEEE Access, 2020.

6.2 Simulation a Evenements Discrets
--------------------------------------
    PRINCIPE : Le temps n'avance pas en continu mais d'evenement en evenement.

    Temps simule :  0.000s    0.001s    0.002s    0.003s    ...
                      |         |         |         |
    Evenements:   [Paquet1   [Paquet1   [Paquet2  [Paquet1
                   cree]      recu       cree]     traite
                              gNodeB]              par UPF]

    Le simulateur :
    1. Prend l'evenement le plus proche dans le temps
    2. L'execute (peut creer de nouveaux evenements)
    3. Avance l'horloge simulee
    4. Repete jusqu'a la fin

    AVANTAGE : On peut simuler 1 heure de reseau en ~10-30 minutes

6.3 Architecture du Reseau Simule
-----------------------------------
Notre reseau simule reproduit une architecture 5G complete selon le
standard 3GPP TS 23.501 :

    TERMINAUX           ACCES RADIO        COEUR 5G          SERVEURS
    (User Equipment)    (RAN)              (5G Core)         (Data Network)

    +----------+
    | ueEmbb   |--+        +---------+      +------+  +-----+  +----------+
    | (video)  |  |        |         |      |      |  |     |  |serverEmbb|
    +----------+  |        |         |      |      |  |     |  +----------+
                  |  radio |         |  N3  |      |N9|     |N6 +----------+
    +----------+  +------->| gNodeB  |----->| iUpf |--->| upf |--->|serverUrllc
    | ueUrllc  |--+   5G   | (tour   | GTP  |(edge)| GTP|(anc)| IP +----------+
    | (gaming) |  |   NR   |  5G)    |      |      |  |     |  +----------+
    +----------+  |        |         |      |      |  |     |  |serverMmtc|
                  |        |         |      +------+  +-----+  +----------+
    +----------+  |        |         |          |        |      +----------+
    | ueMmtc   |--+        |         |          v        v      |serverBE  |
    | (IoT)    |  |        +---------+      +--------+  +---+   +----------+
    +----------+  |             |            | Router |--| R |-------->
                  |             v            +--------+  +---+
    +----------+  |        +---------+
    |ueBestEff |--+        |Scheduler|
    | (web)    |           |  MAC    |
    +----------+           +---------+

    Composants :
    - gNodeB : L'antenne 5G. Gere l'interface radio, le scheduling MAC
      (qui transmet quand et combien), les protocoles radio (PDCP, RLC, MAC, PHY)
    - iUpf (intermediate UPF) : Routeur edge, proche de l'antenne
    - upf (anchor UPF) : Routeur central, point d'ancrage mobilite
    - Les paquets transitent dans des tunnels GTP-U entre gNodeB et UPF

    Configuration radio :
    - Frequence : 3.5 GHz (bande n78, standard 5G mid-band)
    - Bande passante : 100 MHz (50 Resource Blocks)
    - Puissance gNodeB : 40 dBm (~10W), UE : 26 dBm (~400mW)
    - Scheduler : MAXCI (Maximum Carrier to Interference)
    - Zone : 1000m x 1000m, gNodeB au centre (500m, 500m)

    Allocation des ressources par slice :
    +----------+---------+----------+
    | Slice    | % RBs   | Priorite |
    +----------+---------+----------+
    | eMBB     | 40%     | 3 (basse)|
    | URLLC    | 35%     | 1 (haute)|
    | mMTC     | 15%     | 4        |
    | BestEffort| 10%    | 5 (min)  |
    +----------+---------+----------+

6.4 Mesure de Latence dans simu5G
-----------------------------------
La difference fondamentale avec le M/M/1 est la MESURE REELLE de latence.

    Parcours d'un paquet (simplifie) :
    1. CbrSender cree un paquet avec timestamp = t0
    2. Le paquet traverse la pile radio (PDCP -> RLC -> MAC -> PHY)
    3. Transmission radio vers le gNodeB (propagation, fading)
    4. gNodeB recoit, demodule, passe a la couche IP
    5. Tunnel GTP-U vers iUpf puis upf
    6. Routage IP vers le serveur destination
    7. CbrReceiver recoit le paquet, calcule : latence = t_now - t0

    Signal enregistre : cbrFrameDelay (latence end-to-end en secondes)
    Cette latence inclut TOUS les delais : radio, scheduling, file d'attente,
    propagation, traitement UPF, routage IP.

    C'est ce qui rend simu5G plus realiste que le M/M/1 :
    la latence n'est pas estimee par une formule, elle est MESUREE.

    +----------------------+-------+-------+-------+------+--------+----------+
    | Normal               |  4    |  4    |  2    |  2   | 1800s  | Baseline |

6.7 Defis Rencontres et Solutions
-----------------------------------

    DEFI 1 : FUITE D'INFORMATION (Label Leakage)
    -----------------------------------------------
    Probleme : Le LSTM atteignait 100% de precision des la 1ere epoch.
    Cause : Les metriques brutes (latence, throughput) encodent DIRECTEMENT
            le label SLA. Si latence > 20ms alors URLLC = viole.
            Le modele n'apprenait pas a PREDIRE, il lisait la reponse.

    Solution : Validation Leave-One-Scenario-Out (LOSO)
    - On entraine sur 10 scenarios et on teste sur 3 scenarios JAMAIS VUS
    - Le modele ne peut plus "tricher" car il doit generaliser a des
      conditions inedites (nombres d'UEs differents, charges differentes)

    DEFI 2 : STABILITE INTRA-SCENARIO
    ------------------------------------
    Probleme : Meme avec des features indirectes (jitter, charge reseau),
    le modele atteignait F1=0.9999 en cross-validation.
    Cause : Chaque scenario produit des metriques quasi-constantes.
            N'IMPORTE quelle feature qui differe entre scenarios
            permet de separer parfaitement les labels.

    Solution : Le split LOSO resout aussi ce probleme car le modele
    doit generaliser a des scenarios avec des distributions differentes.
    Avec LOSO, le F1 chute a ~0.66 = apprentissage REEL.

    DEFI 3 : URLLC - SEUIL DE DECISION INADAPTE
    -----------------------------------------------
    Probleme : URLLC avait F1=0.49 alors que AUC=0.96.
    L'AUC elevee signifie que le modele CLASSAIT bien les exemples
    (les vrais positifs avaient des scores plus eleves que les negatifs),
    mais le seuil par defaut de 0.5 etait inadapte.
    Le modele avait des scores de probabilite tres proches (entre 0.99 et 1.0),
    et beaucoup de negatifs avaient un score > 0.5.

    Solution : Optimisation du seuil de decision par slice.
    En utilisant la courbe precision-recall, nous trouvons le seuil optimal :
    - URLLC : seuil optimal = 0.994 (au lieu de 0.5)
    - eMBB  : seuil optimal = 1.000
    - Avec ces seuils, URLLC passe de F1=0.49 a F1=0.97

    DEFI 4 : mMTC - GENERALISATION INSUFFISANTE
    -----------------------------------------------
    Probleme : mMTC a un AUC=0.34 (pire que le hasard, 0.50).
    Cause : Les violations mMTC sont concentrees dans quelques scenarios
    specifiques (latence juste au-dessus de 8.8ms). Les scenarios de test
    (MediumLoad, ExtremeLoad, RealTraffic) ont seulement 1.08% de violations
    mMTC, vs 57.7% dans l'ensemble d'entrainement.
    Le modele est "inverse" : il predit des violations la ou il n'y en a pas.

    Ce probleme illustre la difficulte de la generalisation inter-scenarios
    pour les slices a faible variabilite.

6.8 Architecture LSTM (Approche B)
------------------------------------
L'architecture est identique a l'Approche A, avec quelques ajustements :

    Entree : 30 pas de temps x 23 features
    (5 metriques de base + 15 derivees + 3 one-hot slices)

    Features de base :
    1. Slice_Throughput_Mbps
    2. Slice_Latency_ms
    3. Slice_Jitter_ms
    4. Slice_Packet_Loss_pct
    5. Slice_Network_Load_pct

    Features derivees (par feature de base) :
    - Moyenne mobile 5 secondes  (tendance court terme)
    - Moyenne mobile 10 secondes (tendance moyen terme)
    - Variation (derivee)        (acceleration du changement)

    One-hot encoding :
    - slice_URLLC, slice_eMBB, slice_mMTC

    Preprocessing :
    - MinMax scaling sur les features continues
    - Les features one-hot ne sont PAS scalees
    - Sequences construites par (Scenario, Slice Type) pour eviter
      la contamination inter-scenarios

    Split des donnees (Leave-One-Scenario-Out) :
    - Train : 10 scenarios (Normal, OverloadEmbb, OverloadUrllc, OverloadAll,
      RealTrafficOverload, TransitionEmbb, TransitionUrllc, StressEmbb,
      HeavyEmbb, CongestionHigh)
    - Test : 3 scenarios (MediumLoad, ExtremeLoad, RealTraffic)

    Le slice Best Effort est filtre du dataset car il n'a pas de SLA
    significatif et fausserait les metriques.

    APPROCHE A (M/M/1) :
    (+) F1 global plus eleve (0.86) grace aux slices deterministes
    (+) Donnees reelles (captures Wireshark)
    (+) Validation temporelle stable (std=0.04)
    (-) F1 gonfle par Best Effort (100%) et mMTC (98%) deterministes

8. CONCLUSIONS ET PERSPECTIVES
================================================================================

8.1 Contributions de ce PFE
-----------------------------
1. Mise en place d'un pipeline complet de prediction SLA :
   Donnees brutes -> Traitement Big Data (Spark) -> Simulation 5G ->
   Feature Engineering -> LSTM -> Prediction 15 secondes a l'avance

2. Comparaison de deux approches de modelisation :
   - M/M/1 (analytique, rapide, approximatif)
   - simu5G (simulation, lent, realiste)

3. Identification et resolution du probleme de label leakage :
   - Le split temporel classique ne suffit pas quand les metriques
     sont stables intra-scenario
   - La validation Leave-One-Scenario-Out (LOSO) est necessaire
     pour evaluer la vraie capacite de generalisation

4. Optimisation des seuils de decision par slice :
   - Un seuil unique (0.5) est inadapte pour un modele multi-slice
   - L'optimisation par courbe precision-recall ameliore
     significativement URLLC (+0.48 de F1)

5. Resultats :
   - eMBB : F1 = 0.9913 (quasi-parfait)
   - URLLC : F1 = 0.9738 (excellent avec seuil optimise)
   - Prediction 15 secondes a l'avance sur des scenarios inedits

8.2 Limites
-----------
1. mMTC : Le modele ne generalise pas pour les violations mMTC
   (F1=0.02). Les violations sont trop concentrees dans certains
   scenarios et le modele ne voit pas assez de variete.

2. Donnees simulees : L'Approche B repose sur des simulations.
   Bien que simu5G soit valide academiquement (IEEE Access 2020),
   les performances sur un vrai reseau 5G pourraient differer.

3. Variance inter-scenarios : La cross-validation montre une forte
   variance (std=0.22), certains folds atteignent F1=0.95 et
   d'autres F1=0.44.

4. Scenario unique : Toutes les simulations utilisent une seule
   cellule (1 gNodeB). Un reseau reel aurait plusieurs cellules,
   du handover, de l'interference inter-cellules.

8.3 Perspectives
-----------------
1. Ameliorer mMTC : Ajouter des scenarios avec des violations mMTC
   intermediaires (latence autour de 8.8ms avec variabilite)

2. Multi-cellules : Simuler un reseau avec plusieurs gNodeBs et
   du handover pour plus de realisme

3. Transfert learning : Entrainer sur simu5G puis affiner (fine-tune)
   sur des donnees reelles d'un operateur

4. Deploiement temps-reel : Integrer le modele LSTM dans un
   controleur SDN/NFV pour la reallocation proactive de ressources

5. Modeles alternatifs : Tester des Transformers (attention mechanism)
   qui pourraient mieux capturer les dependances long terme
